CHAPTER THREE
SYSTEM ANALYSIS AND DESIGN
3.1 Introduction
This chapter presents the system analysis and design of UsafiLink, a proximity-based exhauster service management system. The chapter begins by describing the Agile system development methodology adopted for the project, followed by a feasibility study assessing the economic, technical, and operational viability of the proposed system. It then details the requirements elicitation process, including the data collection instruments used and the sampling approach applied. The findings from data analysis are presented and used to derive functional and non-functional system requirements. The chapter concludes with a comprehensive system design covering the logical architecture, control flow, database structure, and user interface design of UsafiLink.
3.2 System Development Methodology
UsafiLink was developed using the Agile System Development Methodology, specifically the Scrum framework. Agile was selected because the system involves multiple interdependent components — the React and Vite web application frontend, the Django REST Framework backend, the SQLite development database, the Celery asynchronous task queue, and the M-Pesa Daraja API integration — each of which benefits from incremental development, continuous testing, and iterative refinement based on user feedback. Unlike the Waterfall model, which requires all requirements to be fully defined before development begins, Agile accommodates the evolving nature of requirements that is characteristic of systems deployed in informal service sectors such as exhauster service management in Kenya.
The development was organised into two-week sprints, each producing a working and testable increment of the system. Sprint one focused on user authentication, email OTP verification, and two-factor authentication (2FA/TOTP). Sprint two implemented the GPS service request module and the proximity-matching dispatch algorithm with automated driver timeout management via Celery. Sprint three covered M-Pesa STK Push customer payments and B2C driver payout integration. Sprint four developed the driver rating system, transaction audit logging, and the administrative web dashboard. Sprint five was dedicated to integration testing, performance evaluation of the dispatch algorithm, and user acceptance testing. This iterative structure ensured that the proximity-matching algorithm — the core research component of UsafiLink — could be evaluated and refined across successive sprints based on measured wait time and route efficiency data. The sprint schedule and project timeline are presented in Figures 3.1 and 3.2 in Section 3.6.
3.3 Feasibility Study
A feasibility study was conducted to assess the viability of UsafiLink across three dimensions: technical, economic, and operational. The findings confirmed that the project is feasible within the constraints of the eight-month development timeline and the available resources.
3.3.1 Technical Feasibility
The technologies used to develop UsafiLink are mature, well-documented, and freely available. The frontend is built with React and Vite, producing a responsive progressive web application accessible on both desktop browsers and mobile browsers on Android devices, without requiring a native application installation. The backend is built entirely on Django and Django REST Framework (Python). The database used during development and testing is SQLite, with PostgreSQL and PostGIS recommended for production deployment. The Safaricom Daraja API enables both M-Pesa STK Push customer payments and B2C business-to-customer driver payouts. Celery with a Redis broker handles asynchronous task management.
3.3.2 Economic Feasibility
The development of UsafiLink does not require the purchase of proprietary software licences. Django, React, Vite, Celery, Redis, and SQLite are all open-source. The Safaricom Daraja API sandbox is freely available for development and testing. Hosting costs for the pilot can be accommodated on free-tier cloud services. The system is therefore economically feasible.
3.3.3 Operational Feasibility
The target users — exhauster drivers and customers in urban Kenya — demonstrate high rates of smartphone ownership and mobile internet usage. M-Pesa is embedded in daily financial transactions, reducing adoption barriers. The decision to implement UsafiLink as a responsive web application rather than a native Android application is operationally appropriate given storage space and data cost considerations of the target population. The system is therefore operationally feasible within the defined deployment context. A consolidated feasibility summary matrix is presented as Figure 3.3 in Section 3.6.
3.4 Requirements Elicitation
3.4.1 Data Collection
Requirements elicitation for UsafiLink was conducted using two complementary data collection instruments: structured questionnaires and semi-structured interviews. Structured questionnaires were administered to exhauster drivers in person to collect quantitative data on their current service coordination processes, typical response times, frequency of job conflicts, and payment collection challenges. Semi-structured interviews gathered deeper qualitative data on operational workflows, navigation practices, and attitudes towards digital service management tools. Each interview lasted approximately twenty minutes.
3.4.2 Sampling
A purposive sampling technique was employed. The target population consisted of actively operating exhauster drivers in Nairobi and its peri-urban zones. A sample of eight exhauster drivers was selected, representing both independent operators and drivers affiliated with small exhauster service companies. The findings were used to identify recurring operational themes and derive system requirements rather than to produce statistically generalizable results (Creswell, 2014).

3.8.2 Control Flow and Process Design
The core process flow centres on the service request and proximity dispatch cycle. When a customer submits a service request, the browser captures GPS coordinates via the HTML5 Geolocation API and transmits them to the Django backend. The Haversine algorithm queries available drivers, computes distances, and dispatches the nearest driver. A Celery task enforces the ten-second response timeout. Upon completion, M-Pesa STK Push is triggered, the audit log is updated, and the driver's status resets to available.
Figure 3.10 – Detailed Process Flowchart: Dispatch Algorithm
The flowchart below details the logic of the Haversine proximity dispatch algorithm.
● START: Customer submits service request
1
Browser invokes HTML5 Geolocation API → captures customer latitude and longitude
2
POST /api/service-requests/ → transmit GPS coordinates to Django REST API
3
Backend queries Drivers table: SELECT WHERE availability_status = 'available'
4
For each available driver: compute Haversine great-circle distance from customer coordinates to driver coordinates
⬡
[Decision] Are there any available drivers?
4a
→ NO: Return error to customer: 'No drivers available at this time.' Log request as unfulfilled.
5
→ YES: Sort drivers by computed distance (ascending). Select driver with minimum distance.
6
Create Job record (status = pending). Link to ServiceRequest and assigned Driver.
7
Push job notification to assigned driver's web application interface.
8
Simultaneously queue Celery task: auto_reassign_job(job_id) with 10-second countdown.
⬡
[Decision] Does driver respond within 10 seconds?
9a
→ REJECT or TIMEOUT: Celery worker fires; driver marked busy to exclude from next round. Remove driver from available list. Return to Step 3 (next nearest).
9b
→ ACCEPT: Update Job status = accepted. Cancel Celery timeout task. Notify customer of acceptance.
10
Live tracking begins. Customer polls GET /api/driver-location/{job_id}/ every 5 seconds.
11
Driver navigates to customer using embedded Google Maps directions link.
12
Driver completes service → taps 'Mark Job Complete' → Job status = completed.
13
Django triggers M-Pesa STK Push via Daraja API to customer's M-Pesa phone number.
⬡
[Decision] Payment confirmed via Daraja callback?
14a
→ PAYMENT FAILED: Log failure in TransactionLog. Alert admin. Allow retry.
14b
→ PAYMENT CONFIRMED: Log to TransactionLog. Issue digital receipt. Update Job status = paid.
15
Reset Driver availability_status = available. Customer prompted to submit 1–5 star rating.
■ END: Job complete, payment audited, driver available
